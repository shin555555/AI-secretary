# 凛（りん） — AI秘書 要件定義書

**バージョン**: 1.0
**作成日**: 2026-03-17
**対象フェーズ**: MVP → v2段階的拡張
**アプリ種別**: Bot（LINE）+ ローカルバックエンド

---

## 1. プロダクト概要

### 1.1 エレベーターピッチ

就労継続支援B型事業所の管理者が、LINEで話しかけるだけでスケジュール確認・タスク管理・業務リマインドを行える**自分専用AI秘書**。ローカルPCで動作し、個人情報を外部に出さず、月額0円で運用できる。

### 1.2 解決する課題

- **現状の課題**:
  - タスクが多岐にわたり（受給者証管理、個別支援計画、請求、シフト、出欠、記録作成等）、抜け漏れが発生する
  - スケジュール管理が追いつかず、重要な期限を見逃すリスクがある
  - 一人で管理業務と職業支援員を兼務しており、人手不足で業務に追われている
- **ソリューション**:
  - LINEで自然言語指示するだけで、タスク追加・スケジュール確認・リマインド設定が完了
  - 毎朝8:00に「今日の予定＋期限タスク」を自動通知し、抜け漏れを防止
  - 使うほどユーザーの業務パターンを学習し、先回りして提案してくれる秘書に成長

### 1.3 成功指標（KPI）

| 指標 | 目標値 | 計測方法 |
|------|--------|----------|
| タスク忘れ件数 | 月0件 | ユーザー体感 |
| 朝ブリーフィング確認率 | 90%以上 | LINE既読率 |
| 1日あたりの利用回数 | 5回以上 | interaction_logテーブル |
| セットアップから運用開始 | 1週間以内 | 実測 |

## 2. ユーザー定義

### 2.1 ペルソナ

**メインユーザー（唯一のユーザー）**
- 属性：就労継続支援B型事業所の管理者兼職業支援員
- ITリテラシー：AIを活用したアプリ開発経験あり（AIありき）
- 利用場面：事業所での業務中（PC前）、移動中（スマホLINE）、自宅（PC）
- ゴール：タスクの抜け漏れゼロ、業務時間の効率化、精神的余裕の確保
- フラストレーション：多すぎるタスクの管理、期限管理の煩雑さ、一人で全業務をカバーする負担

### 2.2 ユーザージャーニー（コアフロー）

```
[朝] 8:00 LINE通知受信 → 今日の予定・タスク確認
  → [業務中] 思いついたタスクをLINEで即追加
  → [業務中] 「次何やるべき？」で優先タスク提案を受ける
  → [業務中] 「明日の14時に○○さん面談」でカレンダー登録
  → [夕方] 「今日のまとめ」で完了タスク・残タスク確認
```

各ステップで：
- ユーザーの行動：LINEでテキスト送信（1行〜数行）
- 期待する感情：「秘書に任せれば安心」「忘れても大丈夫」
- 離脱ポイントと対策：
  - LLMの応答が遅い → 「少々お待ちください」即時返信 + 結果を後送
  - 意図が伝わらない → 確認質問で意図を特定してから実行

### 3.2 コマンド・インテント一覧

AI秘書は自然言語を理解するため、固定コマンドは不要。ただし以下のインテントに分類して処理する：

| インテント | トリガー例 | 処理内容 |
|-----------|-----------|----------|
| schedule_today | 「今日の予定」「今日のスケジュール」 | Googleカレンダーから当日イベント取得 |
| schedule_week | 「今週の予定」「来週の予定」 | 指定期間のイベント取得 |
| schedule_create | 「明日14時に面談」「毎週月曜14時に支援会議」 | カレンダーにイベント作成（単発 or 繰り返し） |
| task_add | 「タスク追加：報告書作成」「金曜までに〇〇」 | タスクDB登録（単発） |
| task_recurring | 「国保連請求 毎月7日」「年度末書類整理 毎年3月15日」 | 繰り返しタスク登録 |
| task_list | 「タスク一覧」「今日のタスク」「未完了タスク」「ルーティン一覧」 | タスク一覧取得 |
| task_done | 「タスク1完了」「報告書作成 完了」 | タスクステータス更新 |
| task_priority | 「次何やる？」「1時間空いた」 | 優先タスク提案 |
| briefing | 「今日のまとめ」「ブリーフィング」 | 予定+タスクの要約 |
| preference | 「覚えて：〇〇」「設定変更」 | ユーザー好み更新 |
| general | その他の会話・質問 | LLMによる汎用応答 |

### 3.3 各機能の詳細仕様

#### LINE双方向チャット
**目的**: ユーザーがLINEから自然言語でAI秘書に指示を出し、即座に応答を受ける

**処理フロー**:
- LINE Webhook受信 → 署名検証 → インテント分類 → 該当サービス実行 → LINE返信

**ユーザーアクション & システム応答**:
| アクション | システム応答 | エラー時の挙動 |
|-----------|-------------|--------------|
| テキストメッセージ送信 | インテントに応じた応答（2秒以内目標） | 「申し訳ございません、処理中にエラーが発生しました。もう一度お試しください。」 |
| LLM処理に3秒以上 | 「少々お待ちください...」即時返信→結果をPush | — |
| 意図不明なメッセージ | 「〇〇ということでよろしいでしょうか？」確認質問 | — |

#### 朝のブリーフィング（Push通知）
**目的**: 毎朝8:00に今日の予定とタスクを自動通知

**通知フォーマット例**:
```
おはようございます。本日のブリーフィングです。

📅 今日の予定（3件）
• 10:00-11:00 ○○さん面談
• 13:00-14:00 支援会議
• 15:00 自治体訪問

✅ 今日期限のタスク（2件）
• 国保連請求データ提出【緊急】
• 個別支援計画書（田中さん）更新

🔁 今日の定期タスク（1件）
• 国保連請求データ準備（毎月7日）

⚠️ 明日期限のタスク（1件）
• シフト表確定

本日もよろしくお願いいたします。
```

#### タスク管理
**目的**: 思いついた時にLINEで即座にタスク登録、管理

**バリデーション**:
- タイトル: 必須・最大200文字
- 期限: 任意・過去日付は警告
- 優先度: 1（緊急）〜5（低）、未指定時は3

#### 繰り返しタスク（ルーティン）
**目的**: 月次・年次等の定期業務を一度登録すれば、毎回自動でタスクが生成される

**対応する繰り返しパターン**:
| パターン | 入力例 | 動作 |
|---------|--------|------|
| 毎日 | 「出欠記録 毎日」 | 毎日タスク生成 |
| 毎週X曜 | 「週次ミーティング 毎週月曜」 | 毎週月曜にタスク生成 |
| 毎月X日 | 「国保連請求 毎月7日」 | 毎月7日にタスク生成 |
| 隔月 | 「モニタリング 隔月1日」 | 2ヶ月ごとの1日にタスク生成 |
| X月ごと | 「支援計画更新チェック 6ヶ月ごと 4月と10月」 | 指定月にタスク生成 |
| 毎年X月X日 | 「年度末書類整理 毎年3月15日」 | 毎年3月15日にタスク生成 |

**処理フロー**:
```
ユーザー「国保連請求データ準備 毎月7日」
  → インテント: task_recurring
  → LLMで解析: {title: "国保連請求データ準備", rrule: "monthly", day: 7}
  → recurring_tasks テーブルに登録
  → 凛「毎月7日に「国保連請求データ準備」を繰り返しタスクとして登録しました。
  　　期限前にリマインドもお送りしますか？」
```

**自動生成タイミング**:
- APSchedulerが毎日0:00に翌日分の繰り返しタスクをチェック
- 該当日にtasksテーブルに実タスクとして自動INSERT（status: pending）
- 朝ブリーフィングに自動的に含まれる

**管理コマンド**:
- 「ルーティン一覧」→ 登録済みの繰り返しタスクを表示
- 「ルーティン削除：国保連請求」→ 繰り返し登録を解除
- 「ルーティン変更：国保連請求 毎月10日に変更」→ スケジュール変更

**タスク表示フォーマット例**:
```
📋 未完了タスク（5件）

🔴 1. 国保連請求データ提出（期限: 3/10）
🟡 2. 個別支援計画書更新 田中さん（期限: 3/20）
🟡 3. シフト表確定（期限: 3/18）
🟢 4. 見学者対応マニュアル整備（期限: 3/31）
🟢 5. 備品発注リスト作成（期限なし）

完了にするには「タスク1完了」のように送信してください。
```

#### Googleカレンダー連携
**目的**: 自然言語でスケジュールの確認・登録

**単発予定の処理フロー**:
```
ユーザー「明日の14時から15時に○○さんの面談を入れて」
  → インテント: schedule_create
  → LLMで解析: {title: "○○さん面談", start: "2026-03-18T14:00", end: "2026-03-18T15:00", rrule: null}
  → 重複チェック → Googleカレンダーに登録
  → 「明日3月18日 14:00-15:00に「○○さん面談」を登録しました。」
```

**繰り返し予定の処理フロー**:
```
ユーザー「毎週月曜14時から支援会議を登録して」
  → インテント: schedule_create（繰り返しと判定）
  → LLMで解析: {title: "支援会議", time: "14:00", duration: 60, rrule: "FREQ=WEEKLY;BYDAY=MO"}
  → Google Calendar に繰り返しイベント（RRULE付き）で登録
  → 「毎週月曜14:00-15:00「支援会議」を繰り返し予定として登録しました。」
```

**対応する繰り返しパターン（カレンダー予定）**:
| パターン | 入力例 | Google Calendar RRULE |
|---------|--------|----------------------|
| 毎週X曜 | 「毎週月曜14時に支援会議」 | FREQ=WEEKLY;BYDAY=MO |
| 毎月X日 | 「毎月15日に職員会議」 | FREQ=MONTHLY;BYMONTHDAY=15 |
| 毎年X月X日 | 「毎年4月1日に年度始め会議」 | FREQ=YEARLY;BYMONTH=4;BYMONTHDAY=1 |
| 平日毎日 | 「平日の朝9時に朝礼」 | FREQ=WEEKLY;BYDAY=MO,TU,WE,TH,FR |

**繰り返し予定の編集（LINEから）**:
| アクション | 入力例 | 動作 |
|-----------|--------|------|
| 今回のみキャンセル | 「今週の支援会議キャンセル」 | 該当回のみ削除（以降は継続） |
| 以降すべてキャンセル | 「支援会議 来週以降キャンセル」 | 今後の繰り返しをすべて削除 |
| 繰り返し確認 | 「繰り返し予定一覧」 | 登録中の繰り返しイベントを表示 |

> **注**: カレンダー繰り返し予定はGoogle Calendar側で管理されるため、カレンダーアプリからも確認・編集可能。

## 4. データ設計

### 4.1 主要データモデル

```typescript
// タスク
type Task = {
  id: number;              // 自動採番
  title: string;           // タスク名
  description: string | null; // 詳細（任意）
  priority: number;        // 1=緊急 〜 5=低（デフォルト: 3）
  due_date: Date | null;   // 期限
  status: 'pending' | 'in_progress' | 'done'; // ステータス
  category: string | null; // 'admin' | 'support' | 'personal'
  created_at: Date;
  completed_at: Date | null;
};

// 繰り返しタスク（ルーティン定義）
type RecurringTask = {
  id: number;
  title: string;
  description: string | null;
  priority: number;        // デフォルト優先度
  category: string | null;
  rrule: string;           // 'daily' | 'weekly' | 'monthly' | 'bimonthly' | 'yearly' | 'custom'
  day_of_week: number | null;  // 0=月曜〜6=日曜（weekly用）
  day_of_month: number | null; // 1〜31（monthly用）
  months: string | null;       // "4,10"（custom用、対象月をカンマ区切り）
  remind_days_before: number;  // 何日前にリマインド（デフォルト: 1）
  is_active: boolean;
  created_at: Date;
};

// 会話履歴
type Conversation = {
  id: number;
  role: 'user' | 'assistant';
  content: string;
  intent: string | null;   // 分類されたインテント
  timestamp: Date;
};

// ユーザー好み設定
type Preference = {
  key: string;             // 設定キー（例: 'briefing_time'）
  value: string;           // 設定値（例: '08:00'）
  updated_at: Date;
};

// 行動ログ（パーソナライズ用）
type InteractionLog = {
  id: number;
  action_type: string;     // 'task_create' | 'schedule_check' 等
  metadata: object;        // JSON形式の詳細データ
  timestamp: Date;
};

// 利用者情報（v2）
type ClientInfo = {
  id: number;
  name: string;            // 暗号化DB内で保護
  certificate_number: string | null;  // 受給者証番号
  certificate_expiry: Date | null;    // 受給者証有効期限
  support_plan_expiry: Date | null;   // 個別支援計画更新期限
  notes: string | null;
  is_active: boolean;
};
```

### 4.2 データフロー図

```
[LINEメッセージ]
  → [LINE Webhook (FastAPI)]
    → [署名検証]
    → [インテント分類 (LLM)]
    → ┌── schedule系 → [Google Calendar API] → [応答生成]
       ├── task系    → [SQLCipher DB CRUD]  → [応答生成]
       ├── preference → [SQLCipher DB更新]   → [応答生成]
       └── general   → [LLM会話]            → [応答生成]
    → [LINE Reply API]
  → [ユーザーのLINE]

[APScheduler (定期実行)]
  → [Google Calendar API] + [SQLCipher DB]
  → [ブリーフィング/リマインド生成]
  → [LINE Push API]
  → [ユーザーのLINE]
```

### 4.3 状態管理の方針
- **会話コンテキスト**: メモリ上に直近10ターン保持、DBにも永続化
- **タスク状態**: SQLCipher DBで永続管理
- **ユーザー設定**: SQLCipher DBのpreferencesテーブル
- **LLMの状態**: ステートレス（毎回システムプロンプト+コンテキストを送信）

### 4.4 外部連携

| 連携先 | 用途 | プロトコル | 認証方式 |
|--------|------|----------|---------|
| LINE Messaging API | ユーザーとの対話 | HTTPS Webhook + REST | チャネルアクセストークン |
| Google Calendar API | 予定の読み書き | REST | OAuth2（リフレッシュトークン） |
| Ollama | ローカルLLM推論 | HTTP localhost:11434 | なし（ローカル） |
| Google Gemini API | クラウドLLMフォールバック | REST | APIキー |
| Cloudflare Tunnel | Webhook公開 | HTTPS tunnel | Cloudflareアカウント |

## 5. 非機能要件

### 5.1 パフォーマンス

| 指標 | 目標値 |
|------|--------|
| LINE応答時間（ローカルLLM） | 5秒以内 |
| LINE応答時間（Gemini API） | 3秒以内 |
| タスクCRUD操作 | 500ms以内 |
| カレンダー取得 | 2秒以内 |
| 朝ブリーフィング生成 | 10秒以内 |

### 5.2 セキュリティ

- **DB暗号化**: SQLCipher（AES-256）で全データ暗号化。鍵はOSキーリングに保存
- **PIIフィルタ**: 外部API送信前に個人情報を自動検出・プレースホルダ置換
  - 対象: 利用者氏名、受給者証番号、電話番号、住所
  - 方式: DB登録済みの固有名詞 + 正規表現パターンマッチ
  - `[CLIENT_001]` `[CERT_XXXX]` 等に置換し、応答時に復元
- **LINE署名検証**: X-Line-Signature ヘッダのHMAC-SHA256検証（必須）
- **OAuth2トークン**: 暗号化ファイルとして保存、コードにハードコードしない
- **通信**: HTTPS必須（Cloudflare Tunnel経由で自動TLS）
- **アクセス制限**: LINE UserIDベースで自分のみ応答（他ユーザーは無視）

### 5.3 可用性・障害対応

| シナリオ | 挙動 |
|---------|------|
| PC電源OFF | LINE自動応答「現在オフラインです」（LINE公式アカウント設定） |
| Ollama起動失敗 | Gemini APIにフォールバック（PII除去後） |
| Gemini APIエラー | 「一時的に応答できません」メッセージ返信 |
| Google Calendar接続エラー | タスク機能のみ動作、カレンダー機能は一時停止を通知 |
| DB接続エラー | エラーログ記録、ユーザーにシステムエラー通知 |

### 5.4 エラーハンドリング原則

- すべてのエラーはログファイルに記録（ローテーション付き）
- ユーザー向けメッセージは日本語で分かりやすく
- 外部API障害時はグレースフルデグラデーション（使える機能だけ動かす）
- 予期せぬエラー: 「申し訳ございません、予期しないエラーが発生しました」+ ログ記録

## 6. AI秘書キャラクター設計

### 6.1 ペルソナ: 凛（りん）

- **性格**: しっかり者で頼りになる女性秘書。的確で無駄がなく、必要な情報を簡潔に伝える
- **口調**: 丁寧語（ですます調）。ビジネスライクだが冷たくはなく、適度に温かみがある
- **振る舞い**:
  - 聞かれたことに答えるだけでなく、気づいたことは提案する
  - 「明日期限のタスクがございますが、よろしいでしょうか？」
  - 確認が必要な場合は必ず聞き返す（勝手に判断しない）
  - 褒め言葉やねぎらいも適度に入れる「お疲れさまでした」

### 6.2 システムプロンプト（概要）

```
あなたは「凛（りん）」という名前のAI秘書です。
就労継続支援B型事業所の管理者の専属秘書として働いています。

【性格・口調】
- しっかりした女性秘書。丁寧語（ですます調）で話す
- 簡潔で的確な応答を心がける
- 必要に応じて提案や注意喚起を行う

【業務知識】
- 就労継続支援B型事業所の運営に関する基本知識を持つ
- 受給者証、個別支援計画、国保連請求、モニタリング等の用語を理解
- 月末は請求業務、年度末は計画更新が忙しいことを把握

【できること】
- Googleカレンダーの予定確認・登録
- タスクの追加・一覧・完了・優先順位提案
- 朝のブリーフィング・期限リマインド
- 一般的な質問への回答・業務相談

【行動原則】
- 曖昧な指示は確認してから実行する
- 個人情報を含む話題は慎重に扱う
- 重要な操作（削除・変更）は確認を取る
```

## 7. 技術スタック & アーキテクチャ

### 7.1 採用技術

| レイヤー | 技術 | 選定理由 |
|---------|------|---------|
| バックエンド | Python 3.11+ / FastAPI | 非同期対応、LINE SDK対応、開発速度 |
| LLM（ローカル） | Ollama + gemma2:9b-instruct-q5_K_M | 日本語性能良好、32GB RAMで快適動作 |
| LLM（クラウド） | Google Gemini 2.0 Flash | 無料枠（15RPM, 1M tokens/日）で十分 |
| DB | SQLite + SQLCipher | 暗号化、サーバー不要、バックアップ容易 |
| ORM | SQLAlchemy 2.0 | Python標準的ORM |
| LINE連携 | line-bot-sdk v3 (Python) | 公式SDK |
| カレンダー | google-api-python-client | 公式SDK |
| スケジューラ | APScheduler | FastAPIと統合可能 |
| トンネル | Cloudflare Tunnel (cloudflared) | 安定URL、無料、自動TLS |
| 設定管理 | Pydantic Settings | 型安全な設定管理 |
| 鍵管理 | keyring | OSキーリングでDB暗号化鍵を管理 |

1. [LINE Developers Console](https://developers.line.biz/) でプロバイダー作成

## 付録C: リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| pysqlcipher3がWindowsで動かない | DB暗号化不可 | 標準SQLite + Fernet列レベル暗号化にフォールバック |
| OllamaがCPUで遅い（30秒超） | UX低下 | 小型モデル（gemma2:2b）に切替、またはGemini優先に変更 |
| PCスリープでWebhook切断 | メッセージ取りこぼし | Windows電源設定でスリープ無効化、またはクラウド中継サーバ(v2) |
| LINE Push 200通超過 | 通知停止 | バッチ化（朝1通にまとめる）、Push使用量カウンタ実装 |
| Google OAuthトークン失効 | カレンダー停止 | リフレッシュトークン自動更新、失敗時にLINE通知で再認証促す |
| LLMがスケジュール/タスクを幻覚 | 誤情報 | 必ず実データ（API/DB）を取得してからLLMに渡す。LLMの記憶に頼らない |
