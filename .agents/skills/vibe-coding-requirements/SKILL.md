# Vibe Coding Requirements Skill

バイブコーディングで作るアプリが「最高級の評価」を得るための要件定義書を生成するスキル。
実装するAI・開発者が迷わず動けて、かつ見落としゼロの仕様書を作る。

> **対応アプリ種別**: Web SPA / モバイルアプリ / API・バックエンド / Bot（LINE, Discord, Slack等） / CLI / デスクトップ

---

## モード選択

ヒアリング開始前に、ユーザーの温度感を判断する：

- **フルモード**（デフォルト）: 全セクション網羅の完全版要件定義書
- **軽量モード**: ユーザーが「すぐ作りたい」「まず動くものを」と急いでいる場合 → コア機能一覧 + 技術スタック + データモデル + AIプロンプトのみ生成

---

## フェーズ1：ユーザーへのヒアリング

まず以下を会話から引き出す（わかっていることはスキップ）：

### 必須情報
1. **アプリの一言説明** — 何をするアプリ？
2. **アプリの種別** — Web / モバイル / API / Bot / CLI / デスクトップ？
3. **主要ターゲットユーザー** — 誰が使う？（年齢・リテラシー・利用場面）
4. **コアバリュー** — ユーザーにとって一番嬉しいことは何？
5. **技術スタック希望** — React/Next.js/Flutter/Python/Go etc. 未定ならAIに任せる旨を確認
6. **デプロイ先** — Vercel/AWS/GCP/Heroku/iOS/Android/Desktop/CLI
7. **開発フェーズ** — MVP? v1本番？将来的なスケール感は？

### あると深まる情報（任意）
- 競合サービスや参考になるUI/UX
- 既存システムとの連携有無（外部API、DB、メッセージングサービス等）
- マネタイズモデル
- 締め切り・予算感

ヒアリングが終わったら、次のフォーマットで要件定義書を生成する。

## フェーズ2：要件定義書の生成

以下の構成で **Markdownファイル（`requirements.md`）** を生成する。

```
/mnt/user-data/outputs/requirements.md
```

> **注意**: アプリ種別に応じて不要なセクションはスキップし、該当するセクションを追加すること（例：Bot→画面一覧不要、API設計セクション追加）

## 品質の定義（何をもって「最高級」とするか）

要件定義書は以下6軸で評価する：

| 軸 | 説明 |
|----|------|
| **完全性** | Must機能が漏れなく定義されている |
| **明瞭性** | 実装者が判断に迷う余地がない |
| **UX深度** | ユーザーの感情・離脱ポイント・空状態まで考慮 |
| **技術堅牢性** | エラー処理・型安全・パフォーマンス目標が明示 |
| **アクセシビリティ** | WCAG 2.1 AAレベルの要件が含まれる |
| **即実装可能性** | AIプロンプトがそのまま使えるセクションがある |

生成後にこの6軸でセルフチェックし、弱いセクションを補強してから提出する。
