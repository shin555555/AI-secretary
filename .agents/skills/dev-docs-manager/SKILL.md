# dev-docs-manager
 
Claudeが継続的に開発しやすい環境を整えるため、`docs/` ディレクトリに必要なドキュメントを自動生成・維持管理するスキル。
 
## このスキルの目的
 
Claude（およびチームメンバー）が次のセッションでも迷わず開発を続けられるよう、以下を実現する：
 
1. **文脈の永続化** — セッションをまたいでも設計意図・進捗・決定事項が読み取れる
2. **一貫した構造** — どのプロジェクトでも同じ場所を見れば必要な情報が手に入る
3. **自動メンテナンス** — 実装変更に合わせてドキュメントも更新するフローを確立
 
---
 
## ステップ 1: プロジェクトを分析する
 
まずプロジェクトのルートを調べて、どんなプロジェクトか把握する。
 
```bash
# ルートのファイル構成を確認
ls -la
# 既存のdocsがあれば確認
ls -la docs/ 2>/dev/null || echo "docs/ なし"
# package.json / pyproject.toml / Cargo.toml などから技術スタックを確認
cat package.json 2>/dev/null || cat pyproject.toml 2>/dev/null || cat Cargo.toml 2>/dev/null || true
```
 
分析で確認すること：
- プロジェクト種別（Web / API / CLI / ライブラリ / モノレポ など）
- 技術スタック（言語・フレームワーク）
- 既存のREADMEや仕様書の有無
- `docs/` ディレクトリの有無と既存ファイル

## ステップ 2: docs/ ディレクトリ構造を決める
 
プロジェクト種別に応じて、以下の標準構造から必要なファイルを選択する。
 
### 標準ファイル一覧
 
| ファイル | 目的 | 対象プロジェクト |
|---|---|---|
| `docs/overview.md` | プロジェクト概要・目標・スコープ | 全て |
| `docs/architecture.md` | システム設計・コンポーネント構成 | 全て |
| `docs/setup.md` | 環境構築・ローカル実行手順 | 全て |
| `docs/progress.md` | 開発進捗・タスク管理 | 全て |
| `docs/decisions.md` | 設計上の決定事項（ADR）と理由 | 全て |
| `docs/api.md` | APIエンドポイント仕様 | API / Webバックエンド |
| `docs/components.md` | UIコンポーネント仕様 | フロントエンド |
| `docs/data-model.md` | DBスキーマ・データ構造 | DB使用プロジェクト |
| `docs/deployment.md` | デプロイ手順・インフラ構成 | 本番環境あり |
 
**最小セット（どんなプロジェクトでも必ず作成）**：
- `docs/overview.md`
- `docs/architecture.md`
- `docs/progress.md`

## ステップ 4: 既存ファイルの更新ルール
 
既存の `docs/` ファイルがある場合：
 
1. **上書き禁止** — 既存の内容を消さずに追記・マージする
2. **progress.md は毎回更新** — 作業終了時に完了タスクにチェックを入れ、次のToDoを記録する
3. **decisions.md に追記** — 実装中に設計判断が発生したら即記録する
4. **architecture.md は構造変更時のみ** — 大きな設計変更があったときだけ更新する

## ステップ 5: 完了報告
 
ファイル作成後、ユーザーに以下を報告する：
 
```
✅ docs/ を整備しました
 
作成・更新したファイル:
- docs/overview.md   — プロジェクト概要
- docs/architecture.md — システム設計
- docs/progress.md   — 開発進捗
 
📝 次のセッション開始時は docs/progress.md の「次のセッションでやること」から始めてください。
```

## 運用ルール（Claudeへの指示）
 
- セッション開始時に `docs/progress.md` を確認し、前回の続きから開始する
- 大きな実装変更後は `docs/progress.md` と必要に応じて `docs/architecture.md` を更新する
- 設計上の判断をした場合は `docs/decisions.md` に即座に記録する
- セッション終了前に `docs/progress.md` の「次のセッションでやること」を更新する
- `session-onboarding` スキルや `doc-sync` スキルがある場合はそれらと連携する

## よくある質問
 
**Q: docs/ の内容はどれくらい詳細にすべきか？**  
A: 「3ヶ月後の自分（または別のClaudeセッション）が読んで、すぐ作業を再開できる粒度」を目安にする。過度に詳細にせず、判断の理由と現在の状態に集中する。
 
**Q: 小さなプロジェクトでも全ファイル作るべきか？**  
A: 最小セット（overview / architecture / progress）だけ作れば十分。必要になったら追加する。
 
**Q: コードと docs が乖離したらどうするか？**  
A: `doc-sync` スキルがあればそれを使う。なければ `progress.md` にズレを記録して次の更新機会を待つ。
