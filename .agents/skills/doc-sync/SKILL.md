# Doc-Sync スキル

## 概要

**実装済みコード（Ground Truth）** と **プロジェクトドキュメント** の乖離を自動検出し、ドキュメントを現実に合わせて更新するワークフローです。

どんなプログラミング言語・フレームワーク・プロジェクト構造にも対応します。

---

## 実行ワークフロー

### Step 1: ドキュメントファイルの検出

プロジェクトルートからドキュメントを検索・読み込む（Globツールを使用）：

**優先度高（必ず読む）:**
- `task.md`, `tasks.md`, `TODO.md`, `todo.md`
- `PROGRESS.md`, `progress.md`, `開発進捗.md`, `進捗.md`
- `implementation_plan.md`, `IMPLEMENTATION.md`, `実装計画.md`
- `ROADMAP.md`, `roadmap.md`
- `CLAUDE.md`（プロジェクトルールと現状が記録されている場合）
- `CHANGELOG.md`（未記載の変更があるか確認用）

**優先度中（存在すれば読む）:**
- `docs/` ディレクトリ配下の `.md` ファイル
- `README.md`（実装状況の記述がある場合）
- `sprint.md`, `backlog.md`, `issues.md`

未チェック（`[ ]` または `* [ ]`）のタスク項目を全て抽出してリストアップする。

---

### Step 2: git差分による実装状況の把握（任意）

Gitリポジトリの場合、最近のコミット履歴を参照して実装済み機能を把握する：

```bash
git log --oneline -20
git diff HEAD~5 --name-only
```

これにより「どのファイルが最近変更されたか」をチェックボックス更新の補助情報として使う。

---

### Step 3: 実装ファイルの確認

未チェック項目それぞれについて、対応する実装が存在するか検証する。

**確認方法（言語・プロジェクト種別に応じて選択）:**

| 確認対象 | 確認方法 |
|----------|----------|
| ファイル存在 | Glob/Readツールでパス確認 |
| 関数・クラス実装 | Grepで `def \|class \|function \|const \|export` 検索 |
| APIエンドポイント | ルーティングファイルをGrep |
| UIコンポーネント | コンポーネントファイルの存在確認 |
| DB定義 | マイグレーションファイルの存在確認 |
| 設定・環境変数 | 設定ファイルのキー確認 |

**判定基準:**
- ✅ **実装済み**: ファイルが存在し、対象の関数/コンポーネント/ルートが確認できる
- ⚠️ **部分実装**: ファイルは存在するが、該当機能が見当たらない or スタブ状態
- ❌ **未実装**: 関連ファイルが存在しない

---

### Step 4: チェックボックスの更新

Step 3 で **実装済み** と判定されたタスクのチェックボックスを更新する。

**更新ルール:**
- `[ ]` → `[x]` に変更（完了確認できたもの）
- `[ ]` のまま維持（未実装・部分実装のもの）
- 既存の `[x]` は絶対に変更しない（ユーザーが明示的に戻す指示をした場合のみ変更）

**更新時の注意:**
- 1ファイルずつ確実に更新する（Editツールを使用、まとめて書き換えない）
- 曖昧な場合はユーザーに確認してから更新

---

### Step 5: 追記漏れの補完

ドキュメントに記録されていない実装上の決定事項を補完する：

**補完対象の例:**
- 新たに追加されたファイル・モジュールの説明
- 実装中に変更されたアーキテクチャの決定
- 追加された依存ライブラリ・ツール
- 発見・対処したバグとその解決策
- 次回セッションへの申し送り事項

補完先は内容に応じて適切なドキュメントを選ぶ（進捗記録 / 実装計画 / README など）。

---

### Step 6: 同期完了レポートの出力

```
## ドキュメント同期完了 ✅

### 更新したチェックボックス
- [x] ○○機能の実装（task.md, 行XX）
- [x] △△APIエンドポイントの追加（task.md, 行XX）

### 変更なし（実装未確認のため保留）
- [ ] □□機能（対応ファイルが見つからず）

### 追記した内容
- 開発進捗.md: △△の実装方針を追記
- README.md: セットアップ手順を更新

### 次回セッションへの申し送り
- ◇◇機能が途中のため、○○ファイルから再開予定
```

## エッジケース対応

### ドキュメントが見つからない場合
- `**/*.md` パターンで全mdファイルを列挙してユーザーに選択を促す
- または直接ユーザーにドキュメントパスを確認する

### チェックボックスの確信が持てない場合
- 部分実装の場合は `[-]`（進行中）を使うか、コメントを追記するかをユーザーに確認
- 判断が難しい項目はリストアップしてユーザーに委ねる
- **絶対に確信なしに `[x]` にしない**

### 複数ドキュメント間で記述が矛盾している場合
- 矛盾している箇所をユーザーに提示する
- コードの実装状態を優先的なGround Truthとして扱う
- どちらの記述が正しいかユーザーに確認してから更新する

### チェックボックス書式のバリエーション
- `- [ ]`, `* [ ]`, `+ [ ]` など全パターンを対象とする
- 更新時は既存の書式（`-` or `*` or `+`）を維持する

### 大規模プロジェクトの場合
- タスク数が多い場合は、セクション単位で区切って段階的に処理
- 優先度の高いタスク（直近の sprint / マイルストーン）から処理
- 一度に処理するのは最大30タスクを目安とする

### ドキュメントが複数言語混在の場合（日本語+英語など）
- 既存の言語スタイルを維持して追記する
- 新規追記は既存ドキュメントの言語に合わせる

### ユーザーがチェックを戻したい場合（退行・リバート）
- `[x]` → `[ ]` への変更はユーザーの明示的な指示がある場合のみ実行する
- 対象タスクを確認してから変更する

---

## このスキルを使わないケース

- ドキュメントを新規作成したい → `doc-updater` や直接編集を使う
- AIのコンテキストにプロジェクト情報を読み込みたい → `session-onboarding` を使う
- コードの内容についての質問 → 通常の会話で対応
- ドキュメントが存在しないプロジェクト → まずドキュメント作成を提案する

---

## 使用例

**典型的なトリガー:**
- 「セッション終わりにdoc同期お願い」
- 「task.mdを最新の実装に合わせて更新して」
- 「実装したけどドキュメントが古いままなので整理して」
- `/sync`
- 「次回セッションのために進捗を整理しておいて」
- "sync the docs with what's implemented"
- "mark the completed tasks"

**このスキルが解決する問題:**
1. 「実装したのに `task.md` のチェックを入れ忘れた」
2. 「どこまで実装したか次回セッションで分からなくなる」
3. 「ドキュメントと実際のコードが乖離していて混乱する」
4. `/onboard` コマンドで古い情報を読み込んでしまう
5. 「複数のドキュメントがバラバラな状態になっている」
