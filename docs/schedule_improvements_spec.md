# カレンダー・スケジュール機能 改修設計書

**対象作業者**: Claude Opus (実装担当)
**目的**: ユーザーからのフィードバックに基づき、カレンダー機能のUXを改善し、バグを修正する。

この仕様書では、以下の3つの改修を行います。それぞれの背景・原因・具体的な修正指示を記載しています。順番に実装を進めてください。

---

## 【改修1】 リッチメニュー「予定を追加」時の聞き返し改善

### 1-1. 背景・課題
ユーザーがLINEリッチメニューから「予定を追加」ボタンを押した際、現状は `schedule_create` インテントに分類されますが、メッセージ内に日時が含まれていないため、`app/services/secretary.py` の `_handle_schedule_create` メソッドがエラーとして「日時を読み取れませんでした」と返してしまいます。
人間らしい秘書としては、「予定ですね。いつのどんな予定ですか？」と聞き返すのが自然です。

### 1-2. 修正箇所と実装指示
**対象ファイル**: `app/services/secretary.py`
**メソッド**: `_handle_schedule_create(self, user_message: str)`

1. メソッドの冒頭で、`user_message` が「予定を追加」や「予定を追加したい」など、リッチメニューからの入力や短すぎる抽象的な指示であるかを判定する処理を追加してください。
2. 該当する場合は、LLMでのパース (`parse_schedule_from_message`) を行う前に早期リターンし、以下のような聞き返しのメッセージを返してください。
   * 返信例：「予定の登録ですね。どのような予定を追加しますか？\n（例：「明日14時から1時間、田中さんと面談」「毎週月曜10時に朝礼」）」
3. それ以外にも `parsed.get("start_datetime")` が取得できなかった場合のエラーメッセージを、「日時を読み取れませんでした。具体的な日時を含めて教えてください。」から、「日時の指定が見当たりません。いつの予定か教えていただけますか？」のように少し柔らかいニュアンスに変更してください。

## 【改修2】 カレンダー重複チェック（ダブルブッキング検知）のバグ修正

### 2-1. 背景・課題
ユーザーがわざと時間が被る予定（例：16:00〜）を追加した際、重複チェックで正しく時間が被っている予定を検知できず、終日の予定しか引火しませんでした。
**原因**: `app/services/calendar_service.py` 内の Google Calendar API 呼び出し（`_check_conflicts`, `_get_events_between`, `_get_events_for_range`）において、`timeMin` と `timeMax` にISO形式の文字列を渡す際、**JST（日本時間）のdatetimeをそのまま`.isoformat() + "Z"`としてUTC扱いで送信してしまっている重大なバグ**があります。
これにより、APIへの問い合わせ時間が日本時間から9時間ずれた未来になっており、時間指定の予定がことごとく検索範囲外に漏れていました（終日予定は24時間あるため偶然ヒットしていた）。

### 2-2. 修正箇所と実装指示
**対象ファイル**: `app/services/calendar_service.py`
**メソッド**: 
- `_check_conflicts`
- `_get_events_between`
- `_get_events_for_range`

1. `timeMin` と `timeMax` に渡す `start`と`end`（または `target`等）の datetime オブジェクトについて、正しくタイムゾーン情報を含めてフォーマットするように修正してください。
2. もし `datetime` がタイムゾーンを持たない(naive)場合は、`Asia/Tokyo` などの意図したタイムゾーンを付与した上でISOフォーマットにするか、明示的に `+09:00` を付加してください。
   * 修正例1 (timezone awareにしてisoformat): `start.replace(tzinfo=tz).isoformat()` -> `"2026-03-19T16:00:00+09:00"` (末尾の"Z"はつけない)
   * 修正例2 (文字列付加): `start.isoformat() + "+09:00"` （Python標準のままだとnaiveなのでこれが一番手軽で確実です。既存の `"Z"` を `"+09:00"` に変更するだけで解決します。日本の事業所専用アプリのためこれで問題ありません）
3. この日時シフトのバグはカレンダー取得全般に悪影響を及ぼしているため、上記3つのメソッドの `.list(...)` 呼び出し部分の `timeMin`, `timeMax` のフォーマットをすべて修正してください。

## 【改修3】 「過去・未来を含めた予定の検索」機能の追加

### 3-1. 背景・課題
現状のAIのインテントには「今後のスケジュールの一覧」と「スケジュールの作成」しかなく、「〇〇さんとの打ち合わせっていつだっけ？」というような**特定のキーワードによる予定の検索（過去・未来問わず）**ができません。作成指示と誤認されてしまいます。

### 3-2. 修正箇所と実装指示

#### ① カレンダー検索機能の追加
**対象ファイル**: `app/services/calendar_service.py`
1. キーワードで予定を検索するメソッド `search_events(self, query: str) -> list[dict]` を新規追加してください。
2. 検索範囲は「過去半年」から「未来半年程度」などの `timeMin` と `timeMax` を設定し、`q=query` を引数に渡して `events().list()` を呼び出します（※ここでもタイムゾーンの "+09:00" に注意してください）。
3. 結果を `_parse_events` で整形して返してください。

#### ② インテント分類の拡充
**対象ファイル**: `app/prompts/intent_classifier.py`
1. インテント一覧に以下を追加してください。
   `- schedule_search: 特定の予定を検索する（例: 「〇〇の予定っていつ？」「過去の〇〇の会議」「〇〇さんとの打ち合わせ」）`
2. `VALID_INTENTS` リストにも `"schedule_search"` を追加してください。

#### ③ セクレタリーのルーティング・応答生成
**対象ファイル**: `app/services/secretary.py`
1. インテントのルーティング (`handler_map`) に `"schedule_search": lambda: self._handle_schedule_search(user_message)` を追加してください。
2. メソッド `_handle_schedule_search(self, user_message: str) -> str` を新規実装してください。
3. 実装のロジック:
   - `user_message` から検索キーワードを抽出するためにLLMを呼び出します。
     （プロンプト例：「ユーザーのメッセージから、検索したい予定のキーワード（人名や会議名など）だけを抽出して単語で返してください。メッセージ：{user_message}」）
   - 抽出したキーワードを使って `calendar_service.search_events(keyword)` を呼び出します。
   - 取得できたイベント情報を `calendar_service.format_events_for_display(events, show_date=True)` 等でフォーマットし、「〇〇に関する予定はこちらです」と返答します。
   - 見つからない場合は「〇〇に関する予定は見つかりませんでした。」と返します。
